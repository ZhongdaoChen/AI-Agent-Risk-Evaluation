## 1. Agent能力/爆炸半径分析

**目标**：枚举这个 AI Agent 在宿主系统或外部服务上**能做什么**，而不是找代码漏洞。核心问题是"爆炸半径（Blast Radius）"——一旦 Agent 被滥用或失控，能造成多大影响。

### 数据来源

- GitHub API 获取文件树
- 优先扫描含以下关键词的源文件：`tool`、`agent`、`skill`、`action`、`executor`、`capability`、`function`、`plugin`、`run`、`main`、`workflow`、`command`、`bash`、`shell`、`email`、`browser`、`database`、`cloud`、`aws`
- 同时读取 README.md（了解文档中描述的能力）
- 最多扫描 15 个文件，每文件截取前 400 行
- 全部文件内容合并后发送给 **Qwen qwen-plus** LLM 进行整体分析

### LLM 分析逻辑

LLM 被指示扮演"爆炸半径分析师"角色，要求：
- 只报告有实际代码证据支撑的能力（工具定义、函数调用、导入、配置）
- 每种能力只报告一次（选最清晰的代码证据）
- 说明具体的影响范围（如"可读取任意路径的文件"比"读取文件"更精确）
- 区分能力是否由 LLM 动态管控：只有 LLM 可触发且 LLM 输出会动态影响命令、路径、URL、SQL、请求体、云资源等参数/目标的能力，才计入爆炸半径。
- 参数固定、硬编码、或被安全枚举/白名单约束且 LLM 无法扩展的能力，会作为"参数固定/受限的 LLM 可触发能力"展示，但不扣分。
- 非 LLM 可触发的确定性能力会单独折叠展示，不扣分。

### 能力分类与评分扣除

| 类别 | 含义 | 扣分 |
|---|---|---|
| 💻 终端 / Shell 执行 | Agent 可运行任意 shell/bash/cmd 命令 | −25 |
| ⚙️ 代码解释器 / REPL | 拥有 Python/JS 代码解释器或 REPL 沙箱 | −22 |
| 🖥️ 计算机 / 桌面控制 | 可控制鼠标、键盘、截图、操作 GUI | −22 |
| ✏️ 文件系统写入 / 删除 | 可创建、修改、删除文件或目录 | −18 |
| 🗄️ 数据库写入 | 可执行 INSERT / UPDATE / DELETE | −18 |
| 📧 邮件 / 消息发送 | 可发送邮件、Slack / Teams / Discord 消息 | −15 |
| ☁️ 云服务 | 使用 AWS / GCP / Azure / S3 / Lambda 等 | −15 |
| 🔑 凭证 / 认证访问 | 读取 secrets、OAuth token、API 密钥 | −10 |
| 🌐 外部 API 调用 | 向第三方 HTTP 服务发起请求 | −12 |
| 🌍 网页浏览器自动化 | 打开 URL、抓取网页、填写表单、点击操作 | −12 |
| 📂 文件系统读取 | 读取文件、列出目录 | −8 |
| 🔍 数据库读取 | 执行 SELECT / 查询操作 | −8 |
| 🔎 网络搜索 | 使用 Google / Bing / Tavily / DuckDuckGo 等 | −5 |
| 🧠 长期记忆 / 存储 | 使用向量数据库、持久化记忆存储 | −5 |
| 🔧 其他能力 | 不在以上分类的显著能力 | −5 |

**评分起点：100 分**，每发现一个由 LLM 动态管控的能力类别扣除对应分数（同一类别只扣一次）。参数固定/受限能力和非 LLM 可触发的确定性能力不扣分。

### 评分含义

- **高分（75+）**：Agent 功能简单，只有搜索、只读查询等低影响能力
- **中分（55-74）**：有外部 API 调用、文件读取等中等影响能力
- **低分（35-54）**：有文件写入、数据库操作、云服务等高影响能力
- **极低分（<35）**：拥有 Shell 执行、桌面控制、代码解释器等，几乎可以完全控制宿主系统

## 2. Agent安全护栏

**目标**：评估项目是否实现了有效的 AI 安全控制机制，防止 Agent 失控或被滥用。

### 数据来源

- 优先扫描含 `agent`、`tool`、`safety`、`guardrail`、`prompt`、`llm`、`chain`、`workflow`、`executor`、`planner`、`action` 关键词的源文件
- 读取 README.md（检查安全实践文档）
- 最多扫描 12 个文件，每文件截取前 350 行
- 全部内容合并后发送给 **Qwen qwen-plus** LLM 进行整体分析

### LLM 分析逻辑

LLM 被要求：
- 只有看到**实际运行代码**才报告 POSITIVE（不接受注释、变量名、TODO）
- 为 README 文档评分时，只有详细描述机制实现才加分，仅出现关键词不加分
- 对于未完全实现的机制（如定义了 step limit 但从未强制执行），降为 LOW 并给较低加分

### 评分起点：40 分（保守基准）

AI Agent 天然具有自主性，默认评分保守。

### 检测项目与分值

**正向评分（安全机制存在时加分）**

| 安全机制 | 含义 | 加分 |
|---|---|---|
| 人工审批 / 确认循环 | Agent 在执行高危操作前暂停并请求人工确认 | +15 |
| 步骤 / 轮次 / 迭代上限 | 有明确的 max_steps、max_turns 等硬性限制，防止无限循环 | +15 |
| 输入校验 / 净化 | 对用户输入或外部数据进行检查、清洗再使用 | +10 |
| 输出校验 / 过滤 | 模型回复在执行前经过内容验证 | +10 |
| 工具白名单 / 权限系统 | 明确限制 Agent 可调用的工具集合，而非全部开放 | +10 |
| Prompt Injection 防御 | 有专门的注入检测或防御机制 | +8 |
| 沙箱 / 隔离执行 | Agent 操作在隔离环境中执行 | +8 |
| 请求频率限制 | 工具调用或 API 请求有速率限制 | +5 |
| 操作审计日志 | 记录 Agent 所有操作，支持事后溯源 | +5 |
| README 安全文档 | 文档详细描述安全实践（human-in-the-loop、护栏、负责任使用） | +3 至 +5 |

**负向评分（危险模式存在时扣分）**

| 危险模式 | 含义 | 扣分 |
|---|---|---|
| exec/eval 直接执行 LLM 输出 | 将模型原始输出传入 exec() 或 eval()，可造成任意代码执行 | −25 |
| 用户输入直接注入系统提示 | 用户输入被字符串拼接到 system prompt，存在严重 Prompt Injection | −15 |
| Agent 可执行来自 LLM 输出的任意 Shell 命令 | 无验证地将 LLM 输出传给 shell | −12 |
| 完全未发现安全机制（惩罚项） | 扫描了所有文件，一项安全控制都没有 | −15 |

### LLM 分值限制（防止幻觉通货膨胀）

LLM 提议的 `score_delta` 会被系统强制夹紧：
- POSITIVE：最少 +3，最多 +15
- LOW：−5 至 +5
- MEDIUM：−10 至 −1
- HIGH：−15 至 −8
- CRITICAL：−30 至 −12

## 3. 开源项目声誉与活跃度

**目标**：通过 GitHub 公开数据判断项目的社区信任度、维护状态和安全成熟度。

### 数据来源

- GitHub REST API：`/repos/{owner}/{repo}`
- GitHub Contributors API：`/repos/{owner}/{repo}/contributors`
- GitHub Contents API：检查 `SECURITY.md` 是否存在
- GitHub Contents API：检查 `.github/workflows/` 目录是否存在

### 评分项目（满分 100）

**Stars 数量（0-25 分）**

| Stars 范围 | 得分 | 含义 |
|---|---|---|
| ≥ 10,000 | +25 | 业界广泛认可，经过大量用户验证 |
| 1,000 - 9,999 | +20 | 成熟项目，有一定社区基础 |
| 100 - 999 | +15 | 小众但有受众 |
| 10 - 99 | +10 | 早期项目，验证有限 |
| < 10 | +5 | 知名度极低，⚠️ 触发警告 |

Stars 反映的是**社区验证程度**，不直接等于代码质量，但高 Stars 项目通常经过更多眼睛的审查。

**最后提交时间（0-25 分）**

| 距今天数 | 得分 | 含义 |
|---|---|---|
| ≤ 30 天 | +25 | 活跃维护中 |
| 31 - 90 天 | +20 | 正常维护节奏 |
| 91 - 180 天 | +15 | 维护减慢，但仍有人看 |
| 181 - 365 天 | +8 | ⚠️ 活跃度下降，安全补丁可能滞后 |
| > 365 天 | +0 | 🚫 可能已停止维护，漏洞将无法修复 |

**贡献者数量（0-20 分）**

| 贡献者 | 得分 | 含义 |
|---|---|---|
| ≥ 20 人 | +20 | 多元社区，代码经过多人审查 |
| 10 - 19 人 | +15 | 健康的核心团队 |
| 5 - 9 人 | +10 | 小团队，尚可 |
| 2 - 4 人 | +7 | 人员依赖风险较高 |
| 1 人 | +3 | ⚠️ Bus Factor = 1，核心开发者离开后项目可能无人维护 |

**SECURITY.md（0 或 +15 分）**

检查仓库根目录是否存在 `SECURITY.md`。
- 存在：+15 — 说明项目有正式的漏洞报告渠道和响应流程
- 不存在：+0，触发警告 — 安全研究人员发现漏洞后无处报告

**CI/CD 工作流（0 或 +15 分）**

检查 `.github/workflows/` 目录是否存在且包含文件。
- 存在：+15 — 自动化测试和发布，代码变更经过流程把关
- 不存在：+0，触发警告 — 代码直接合并无自动检验

**惩罚项**

| 情形 | 扣分 | 原因 |
|---|---|---|
| 仓库已归档（Archived） | −30 | 归档项目不接受任何更新，包括安全补丁 |
| 这是一个 Fork | 信息提示 | 需确认与上游是否保持同步，可能遗漏上游安全修复 |

## 4. 依赖包健康度

**目标**：通过 [deps.dev](https://deps.dev) API 评估项目本身（如果它是一个已发布的包）的包级别健康状况，并交叉验证依赖许可证和漏洞。

### 数据来源

- `deps.dev` API：`/v3alpha/projects/{url}`（通过 GitHub URL 查找项目记录）
- `deps.dev` API：`/v3alpha/packages/{system}/{name}`（获取包详情）
- 检测以下生态系统：PyPI（`requirements.txt`、`pyproject.toml`、`setup.py`）、npm（`package.json`）、Go（`go.mod`）、Cargo（`Cargo.toml`）、Maven（`pom.xml`）

### 评分项目（基准 100 分）

**deps.dev 项目记录**

| 状态 | 分值 | 含义 |
|---|---|---|
| 有记录 | +0（不额外加分） | 项目存在于 deps.dev 索引中 |
| 无记录 | −10 | deps.dev 无法验证项目，透明度较低 |

**Open Issues 数量**

| 数量 | 分值 | 含义 |
|---|---|---|
| 积压过多（远超均值） | −8 | Issue 积压严重，可能存在未处理的 bug 和安全问题 |
| 正常范围 | +0 | 无惩罚 |

**包发布状态**

- 已发布到 npm / PyPI 等 → +0（确认存在）
- 未发布 → 信息提示，说明这可能是一个应用而非库

**依赖者数量（Dependents Count）**

| 依赖者数量 | 含义 |
|---|---|
| 大量包依赖此项目 | ⚠️ 高影响力：如果此包被攻击，影响范围极广（供应链风险） |
| 少量 | ℹ️ 影响范围有限 |

此指标反映的是**供应链爆炸半径**，依赖者越多，该包越是攻击者的高价值目标。

**deps.dev 安全公告**

- 有安全公告记录：每条 −10 分
- 无公告：+正向确认

### 注意

deps.dev 的漏洞数据与 OSV.dev 的数据略有不同，两个模块互为补充。

## 5. 依赖漏洞扫描

**目标**：扫描项目所有依赖的已知 CVE 漏洞，通过 [OSV.dev](https://osv.dev) 数据库进行查询。

### 数据来源

- **OSV.dev API**：`https://api.osv.dev/v1/query`（批量查询）
- 解析以下依赖文件：
  - Python：`requirements.txt`、`requirements-dev.txt`、`pyproject.toml`
  - Node.js：`package.json`、`package-lock.json`
  - Go：`go.mod`
  - Rust：`Cargo.toml`

### 扫描流程

1. 解析依赖文件，提取包名和版本号
2. 对每个包向 OSV.dev API 发起查询
3. 获取 CVE ID、CVSS 分数、严重等级
4. 按严重等级排列输出

### 评分扣除（基准 100 分）

| 漏洞等级 | 每条扣分 | 含义 |
|---|---|---|
| CRITICAL | −25 | CVSS ≥ 9.0，可能导致远程代码执行或完全失控 |
| HIGH | −12 | CVSS 7.0-8.9，高风险但利用条件更严格 |
| MEDIUM | −5 | CVSS 4.0-6.9，有条件限制，需关注但不必立即停用 |

无 CVE → +正向确认（所有已检查包均无已知漏洞）

### 输出格式

每条漏洞按以下格式输出：

```
包名：CVE 链接，依赖来源文件
```

例如：
```
requests: https://osv.dev/vulnerability/PYSEC-2023-XXX，requirements.txt
```

### 评分含义

- **高分（85+）**：依赖库基本安全，可正常使用
- **中分（60-84）**：有若干 MEDIUM 漏洞，建议升级
- **低分（35-59）**：存在 HIGH 漏洞，使用前需评估影响面
- **极低分（<35）**：有 CRITICAL 漏洞，强烈建议暂停使用直至修复

## 6. 数据隐私

**目标**：评估项目是否存在用户数据泄露风险，包括对话内容被记录、PII 数据处理不当、第三方追踪等。

### 数据来源

- 检查仓库是否存在隐私文档：`PRIVACY.md`、`DATA_HANDLING.md`、`PRIVACY_POLICY.md`、`docs/privacy.md`
- 正则表达式扫描源代码中的隐私相关模式
- 扫描范围：含 `log`、`user`、`prompt`、`data`、`privacy`、`telemetry`、`track` 关键词的源文件

### 评分起点：70 分（中性基准）

### 隐私文档检查

| 文件 | 加分 | 含义 |
|---|---|---|
| `PRIVACY.md` | +15 | 正式的隐私政策说明 |
| `DATA_HANDLING.md` | +10 | 详细的数据处理说明 |
| `PRIVACY_POLICY.md` | +10 | 隐私政策（另一命名方式） |
| `docs/privacy.md` | +5 | 文档目录中包含隐私说明 |
| 以上均无 | −10 | 无法了解数据处理方式，触发警告 |

### 代码模式检查

**正向模式（加分）**

| 模式 | 加分 | 含义 |
|---|---|---|
| 数据匿名化 / 脱敏 | +10 | 代码含 `anonymize`、`pseudonymize`、`redact`、`mask_pii` 等，保护 PII |
| 遥测默认关闭 | +10 | `TELEMETRY_ENABLED = False`、`disable_telemetry` 等，用户隐私优先 |
| 数据留存策略 | +5 | 有 `data_retention`、`delete_after`、`ttl` 等过期机制 |
| 静态数据加密 | +5 | 存储时使用 AES-256、Fernet 等加密 |

**负向模式（扣分）**

| 模式 | 扣分 | 含义 |
|---|---|---|
| 将用户输入/对话写入日志 | −20 | `logging.info(prompt)` 等，对话内容持久化，GDPR 风险 |
| print 输出用户输入 | −10 | `print(user_input)` 可能被重定向到日志文件 |
| 第三方行为追踪 | −10 | PostHog、Mixpanel、Segment、Amplitude 等 SDK 集成 |
| 遥测默认开启 | −15 | `TELEMETRY_ENABLED = True`，数据收集无需用户同意 |
| 将用户输入写入文件 | −15 | `open(path, 'a').write(user_input)` 等持久化行为 |

## 7. 供应链完整性

**目标**：评估项目的依赖管理和分发机制是否能防止供应链攻击（如依赖替换、模型文件篡改）。

### 数据来源

- 扫描仓库文件树，检查特定文件是否存在
- 解析 `requirements.txt` 计算版本固定率
- 调用 GitHub Releases API 检查发布资产
- 调用 HuggingFace API 检查模型 Card
- 扫描源代码中的模型下载和哈希校验模式

### 评分起点：60 分（中性基准）

### 依赖锁文件（最重要指标，±20 分）

检查以下文件是否存在：`poetry.lock`、`package-lock.json`、`yarn.lock`、`pnpm-lock.yaml`、`Pipfile.lock`、`requirements.lock`、`uv.lock`、`pdm.lock`

| 状态 | 分值 | 含义 |
|---|---|---|
| 存在锁文件 | +20 | 所有环境安装完全相同的依赖版本，防止隐式升级引入恶意代码 |
| 不存在 | −15 | ⚠️ 每次安装可能获取不同版本，攻击者可通过发布恶意新版本实施供应链攻击 |

### requirements.txt 版本固定率

| 固定率 | 分值 | 含义 |
|---|---|---|
| ≥ 90% 使用 `==` | +10 | 依赖版本明确，可复现的构建 |
| < 30% 使用 `==` | −10 | 大量使用 `>=`、`~=` 等宽松约束，版本漂移风险高 |
| 30%-90% | +0 | 部分固定，无特别奖惩 |

### SBOM（软件物料清单）

检查是否存在：`sbom.json`、`sbom.xml`、`cyclonedx.json`、`cyclonedx.xml`、`spdx.json`、`bom.json`、`bom.xml` 或任何路径下含 `sbom`、`cyclonedx`、`spdx` 的文件。

- 存在：+10 — SBOM 提供完整依赖组件清单，便于 CVE 追踪和合规审计
- 不存在：不扣分，但属于安全最佳实践缺失

### GitHub Release 校验文件

检查最新 Release 的 assets 是否包含 SHA256 / checksum / 签名文件（`.sig`）。

- 存在：+10 — 用户可验证下载包的完整性
- 不存在：不扣分

### 模型文件哈希校验（AI Agent 特有）

扫描源代码中是否在下载模型权重后验证哈希：

- 存在 `sha256`、`md5`、`hash_check`、`verify_hash`、`checksum` 等：+5
- 检测到通过 `wget`/`curl`/`requests.get` 下载 `.bin`/`.pt`/`.safetensors`/`.gguf`/`.onnx` 但无哈希校验：−10

### HuggingFace 模型版本固定（AI Agent 特有）

- `from_pretrained(model_id, revision="commit_hash")` → +5（版本可追溯）
- `from_pretrained(model_id)` 无 revision → ℹ️ 信息提示（建议固定）

## 8. 运行时隔离

**目标**：评估 AI Agent 是否被容器化或沙箱化部署，防止其逃逸到宿主环境或获取超出需要的权限。

### 数据来源

- 检查仓库中是否存在 `Dockerfile`（及常见变体）
- 解析 Dockerfile 内容中的安全配置
- 检查 `docker-compose.yml` / `docker-compose.yaml` 及其安全配置
- 检查 `.devcontainer/devcontainer.json`（VS Code Dev Container）
- 扫描源代码和配置中的沙箱平台引用

### 评分起点：50 分（中性基准，因为很多 Agent 项目合理地不包含 Docker）

### Dockerfile 检查

| 检查项 | 分值 | 含义 |
|---|---|---|
| Dockerfile 存在 | +15 | 支持容器化部署，有助于环境隔离 |
| 以非 root 用户运行（`USER nonroot`） | +20 | 遵循最小权限原则，即使容器被攻破也无法获取宿主 root |
| 以 root 运行（`USER root`） | −20 | 容器内进程拥有最高权限，提权风险极高 |
| 未指定 USER（默认 root） | −15 | ⚠️ 默认以 root 运行，建议添加 `USER nonroot` |
| 设置了内存 / CPU 限制 | +5 | 防止 Agent 耗尽宿主资源 |
| 使用了 `--privileged` 标志 | −20 | 🚨 等同于给容器宿主机 root 权限，极度危险 |

### docker-compose 检查

| 检查项 | 分值 | 含义 |
|---|---|---|
| docker-compose 配置存在 | ℹ️ 正向提示 | 一键部署支持 |
| 挂载 Docker socket（`/var/run/docker.sock`） | −20 | 🚨 挂载 Docker socket = 给容器宿主机 root 权限，可完全逃逸容器隔离 |
| `privileged: true` | −20 | 🚨 与 Dockerfile --privileged 同等危险 |
| 设置了资源限制 | +5 | Compose 级别的资源控制 |

### Dev Container 配置

存在 `.devcontainer/devcontainer.json` → +10

说明开发者重视标准化环境，减少"在我电脑上能跑"的安全漏洞。

### 沙箱平台集成（AI Agent 特有）

扫描代码和配置中是否引用以下沙箱平台：`e2b`、`daytona`、`modal`、`fly.io`、`docker`（作为代码执行沙箱）。

- 存在沙箱集成：+10 — 代码执行在独立环境中，大幅降低宿主逃逸风险
- 完全没有任何隔离配置：−10 — ⚠️ Agent 直接在用户宿主机上裸跑，所有能力均在无隔离环境中执行

- 如果仓库不是公开可访问，则认定为公司员工自研项目。扫描仍可继续执行，但以下更偏向公开项目/开源生态的模块不计入最终风险评分：⭐ 开源项目声誉与活跃度、🌐 deps.dev 包健康度、📦 依赖漏洞扫描、⛓️ 供应链完整性、🐳 运行时隔离。
- 被排除模块的结果仍会展示，供人工参考。

## 附录：LLM 使用说明

本平台两个模块（Agent能力/爆炸半径分析、Agent安全护栏）使用 **阿里云 Qwen qwen-plus** 模型进行语义分析，具备以下优势：

- **理解上下文**：不是简单关键词匹配，能理解代码的逻辑意图
- **减少误报**：`max_steps = None` 不会被误认为是步骤限制
- **跨文件关联**：多个文件内容合并发送，能发现跨文件的模式
- **结构化输出**：强制 JSON 输出格式，`temperature=0.05` 确保稳定性

**局限性**：
- 无法检测运行时动态加载的内容（如从 URL 获取的 system prompt）
- 受上下文窗口限制，每文件截取前 350-400 行
- LLM 分析耗时约 60-120 秒，是整个分析流程中最慢的环节
